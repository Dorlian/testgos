## Вопрос 6. Особенности применения CMS, обзор современных CMS. Возможности облачных конструкторов разработки web-приложений. Система контроля версий Git. Современные редакторы кода и IDE для разработки web-приложений

### 1. Системы управления контентом (CMS — Content Management System)
**CMS** — это информационная система или компьютерная программа, используемая для обеспечения и организации совместного процесса создания, редактирования и управления контентом (содержимым) веб-сайта без необходимости написания программного кода с нуля.

* **Особенности применения CMS:**
  * Разделение контента и оформления (контент хранится в БД, визуальное представление задается шаблонами/темами).
  * Наличие интуитивной панели администратора (WYSIWYG-редакторы).
  * Разграничение прав доступа пользователей (роли: суперадминистратор, редактор, автор, подписчик).
  * Расширяемость за счет плагинов и модулей.
* **Обзор современных CMS:**
  * **WordPress:** Самая популярная CMS в мире (занимает более 40% рынка сайтов). Написана на PHP, использует MySQL. Применяется для блогов, визиток, корпоративных сайтов и интернет-магазинов (через плагин WooCommerce). Плюсы: огромное сообщество, миллионы плагинов. Минусы: уязвимости безопасности, низкая скорость работы при избытке плагинов.
  * **Bitrix (1С-Битрикс):** Популярная в РФ коммерческая CMS. Написана на PHP. Отличается тесной интеграцией с учетными системами «1С:Предприятие», складским учетом и кассами. Применяется для крупных интернет-магазинов и корпоративных порталов. Требует высокой квалификации разработчиков.
  * **Joomla! и Drupal:** Универсальные CMS средней популярности. Drupal ориентирован на сложные, высоконагруженные проекты с кастомными типами данных и гибким разграничением прав.

### 2. Облачные конструкторы сайтов (SaaS-платформы)
Представляют собой готовые облачные решения, где разработка ведется в визуальном редакторе методом Drag-and-Drop (перетаскивание блоков), а хостинг и обслуживание берет на себя платформа.
* **Примеры:** Tilda Publishing, Wix, Squarespace, Webflow.
* **Возможности конструкторов:**
  * Быстрый запуск (Time-to-Market) — сайт-визитку или лендинг можно собрать за один день.
  * Zero-Block (в Tilda) / свободное позиционирование (в Webflow) — профессиональный дизайн без кода.
  * Встроенные маркетинговые инструменты: CRM-системы, формы сбора заявок, аналитика, SEO-настройки.
* **Ограничения:** Зависимость от платформы (сложно перенести сайт на свой хостинг), ограниченный доступ к исходному коду, сложность интеграции с нестандартными внешними API, ежемесячная абонентская плата.

### 3. Система контроля версий Git
**Git** — распределенная система контроля версий, предназначенная для отслеживания изменений в файлах и координации совместной работы разработчиков.
* **Ключевые понятия:**
  * *Репозиторий (Repository):* Каталог проекта, где Git хранит всю историю изменений. Бывает локальным (на компьютере разработчика) и удаленным (GitHub, GitLab, Bitbucket).
  * *Коммит (Commit):* Снимок (snapshot) состояния проекта в определенный момент времени с уникальным хэшем (SHA-1).
  * *Ветка (Branch):* Направление разработки, изолированное от основной линии (main/master). Позволяет безопасно создавать новые функции.
  * *Слияние (Merge) и Pull Request (PR):* Процессы интеграции изменений из одной ветки в другую с автоматическим или ручным разрешением конфликтов.
* **Схема жизненного цикла файлов в Git:**
  1. **Untracked** (неотслеживаемые файлы).
  2. **Unmodified** (отслеживаемые, без изменений).
  3. **Modified** (измененные файлы в рабочей директории — Working Directory).
  4. **Staged** (подготовленные для коммита в индексе — Staging Area).

### 4. Современные редакторы кода и IDE
* **Текстовые редакторы (легковесные, расширяемые):**
  * *VS Code (Visual Studio Code):* Бесплатный редактор от Microsoft. Де-факто стандарт веб-разработки благодаря огромной экосистеме расширений, встроенному терминалу, поддержке Git и высокой производительности.
* **Интегрированные среды разработки (IDE — тяжеловесные, "всё из коробки"):**
  * *WebStorm / PhpStorm / PyCharm:* Коммерческие IDE от компании JetBrains. Обладают глубоким пониманием кода, мощным автодополнением (IntelliSense), встроенными дебаггерами, профилировщиками производительности и инструментами работы с БД.

## Вопрос 7. Характеристика, назначение и основные понятия баз данных. Жизненный цикл баз данных. Основные модели данных

### 1. Характеристика, назначение и понятия
**База данных (БД)** — это организованная в соответствии с определенными правилами и поддерживаемая в памяти компьютера совокупность взаимосвязанных данных, характеризующая состояние некоторой предметной области и предназначенная для удовлетворения информационных потребностей пользователей.
* **Назначение:** Обеспечение централизованного, надежного, непротиворечивого и быстрого хранения, поиска и модификации информации, исключение избыточности данных.
* **Основные понятия:**
  * *Предметная область (ПО):* Часть реального мира, объекты и процессы которой моделируются в БД (например, «Университет», «Банк»).
  * *Система управления базами данных (СУБД):* Комплекс программных и языковых средств, необходимых для создания БД, поддержания их в актуальном состоянии и организации доступа к ним пользователей.
  * *Схема БД:* Метаданные, описывающие структуру хранения (таблицы, связи, типы полей, ограничения).
  * *Транзакция:* Логическая единица работы с данными, состоящая из последовательности операций чтения/записи, выполняющаяся по принципу «всё или ничего» (свойства **ACID**: Atomicity, Consistency, Isolation, Durability).

### 2. Жизненный цикл базы данных (DBLC — Database Life Cycle)
Жизненный цикл БД тесно связан с жизненным циклом всей информационной системы и состоит из следующих фаз:
1. **Планирование и анализ требований:** Определение целей создания БД, изучение предметной области, интервьюирование пользователей, сбор требований к данным, производительности и безопасности.
2. **Проектирование БД:**
   * *Концептуальное проектирование:* Построение независимой от СУБД ER-модели (сущность-связь).
   * *Логическое проектирование:* Преобразование ER-модели в реляционные отношения (таблицы), проведение нормализации.
   * *Физическое проектирование:* Выбор конкретной СУБД, определение типов данных полей, создание индексов, настройка параметров хранения на диске.
3. **Реализация (наполнение):** Создание структуры БД на сервере с помощью DDL-запросов (SQL), миграция первичных данных, разработка процедур и триггеров.
4. **Тестирование и отладка:** Проверка целостности связей, тестирование производительности под нагрузкой, аудит безопасности.
5. **Эксплуатация и поддержка:** Регулярное резервное копирование, оптимизация медленных запросов, обновление схем данных при расширении функционала ИС.
6. **Устаревание и вывод из эксплуатации:** Миграция данных в новые системы.

### 3. Основные модели данных
Модель данных определяет структуру данных, правила манипулирования ими и ограничения целостности.
* **Иерархическая модель:** Данные представляются в виде дерева (древовидной структуры). Существует один корневой узел, остальные связаны отношением «предок-потомок» (один ко многим). Пример: файловая система диска, документы XML/JSON.
* **Сетевая модель:** Расширение иерархической модели, где потомок может иметь более одного предка (реализует связь «многие ко многим»). Сложна в администрировании и запросах.
* **Реляционная модель:** Разработана Эдгаром Коддом в 1970 году. Данные представляются в виде совокупности взаимосвязанных двумерных таблиц (отношений). Каждая строка — это кортеж (запись), каждый столбец — атрибут (поле).
* **Постреляционные и NoSQL модели (для Big Data и неструктурированных данных):**
  * *Документо-ориентированная:* Хранение данных в виде иерархических документов (JSON/BSON). Пример: MongoDB.
  * *Ключ-значение (Key-Value):* Сверхбыстрые ассоциативные массивы в оперативной памяти. Пример: Redis.
  * *Графовая:* Данные хранятся в виде узлов (вершин) и ребер (связей с атрибутами). Идеально для соцсетей. Пример: Neo4j.

## Вопрос 8. Трехуровневая архитектура описания данных ANSI/X3/SPARC. Уровни абстракции. Концепции многоуровневой информационной архитектуры

В 1975 году комитет стандартов ANSI/X3/SPARC предложил трехуровневую архитектуру СУБД для обеспечения главного принципа БД — **независимости данных**.

```
[ Пользователи / Приложения ]   <--- (Внешние представления)
         |
 +-------v---------------------+
 |   Внешний уровень (External) |  <--- Индивидуальные представления пользователей
 +-------+---------------------+
         |  (Отображение Внешний-Концептуальный)
 +-------v---------------------+
 | Концептуальный (Conceptual) |  <--- Общее логическое представление всей БД
 +-------+---------------------+
         |  (Отображение Концептуальный-Внутренний)
 +-------v---------------------+
 |   Внутренний (Internal)     |  <--- Физическое представление на диске
 +-------+---------------------+
         |
[ Физическая память / Файлы ]
```

### 1. Три уровня абстракции ANSI/X3/SPARC
1. **Внешний уровень (External Level):**
   * *Сущность:* Уровень представлений (представления конечных пользователей и прикладных программ).
   * *Описание:* Каждый пользователь видит только те данные, которые необходимы для его работы (например, бухгалтер видит оклады сотрудников, а менеджер по кадрам — только анкетные данные).
   * *Реализация:* В реляционных СУБД реализуется через механизмы представлений (VIEW) и ограничения прав доступа.
2. **Концептуальный уровень (Conceptual Level):**
   * *Сущность:* Единое интегрированное логическое описание всей базы данных, свободное от деталей физического хранения.
   * *Описание:* Содержит информацию о всех сущностях предметной области, их атрибутах, связях, а также бизнес-правилах и ограничениях целостности. Этот уровень проектируется администратором БД.
   * *Реализация:* Описание таблиц, внешних ключей, проверочных ограничений (CHECK).
3. **Внутренний (внутренне-физический) уровень (Internal Level):**
   * *Сущность:* Физическое описание структуры хранения данных в СУБД на запоминающих устройствах.
   * *Описание:* Описывает, как именно записи распределяются по блокам данных на диске, какие структуры данных используются для хранения файлов (B-деревья, хэш-таблицы), как организованы индексы для ускорения поиска, как выделяется память.
   * *Реализация:* Файловая структура табличных пространств (Tablespaces) СУБД, индексы.

### 2. Принцип независимости данных
Независимость данных означает возможность изменения описания на более низком уровне абстракции без необходимости корректировки описаний на более высоких уровнях.
* **Логическая независимость данных:** Возможность изменения концептуального уровня (добавление новой таблицы, нового столбца в существующую таблицу) без изменения внешних представлений (VIEW) и переписывания прикладного кода приложений.
* **Физическая независимость данных:** Возможность изменения физического уровня хранения (перенос файлов БД на другой диск, смена типа индекса с B-tree на Hash, разбиение таблицы на секции — partitioning) без изменения концептуальной схемы БД.

## Вопрос 9. Классификация систем управления базами данных (СУБД). Основные объекты и их назначение в реляционной СУБД

### 1. Классификация СУБД
* **По используемой модели данных (архитектуре):**
  * *Реляционные (RDBMS):* Работают со строгими табличными структурами (PostgreSQL, MySQL, Oracle, Microsoft SQL Server).
  * *Нереляционные (NoSQL):* Документные, графовые, ключ-значение, колоночные (Cassandra, MongoDB, Redis, ClickHouse).
  * *Объектно-ориентированные и объектно-реляционные:* Расширяют реляционную модель поддержкой пользовательских типов данных и наследования классов (PostgreSQL частично поддерживает эти концепции).
* **По степени распределенности:**
  * *Локальные (централизованные):* Вся СУБД и файлы данных находятся на одном компьютере/сервере.
  * *Распределенные:* Части БД размещаются на различных узлах сети под управлением единой СУБД (реализуется через репликацию, шардирование и распределенные транзакции).
* **По способу хранения данных:**
  * *Дисковые:* Хранят основные данные на жестких дисках/SSD, кэшируя часто используемые страницы в ОЗУ.
  * *In-Memory (Резидентные):* Хранят все данные в оперативной памяти для обеспечения максимального быстродействия (Redis, Memcached, SAP HANA).
* **По сфере применения и типу нагрузки:**
  * *OLTP (Online Transaction Processing):* Оптимизированы для обработки огромного количества коротких транзакций записи/чтения в реальном времени (например, банковские переводы, заказы в интернет-магазине). Пример: PostgreSQL.
  * *OLAP (Online Analytical Processing):* Предназначены для выполнения сложных аналитических запросов над огромными массивами исторических данных (построение отчетов, агрегации). Пример: ClickHouse, Snowflake.

### 2. Основные объекты реляционной СУБД и их назначение
1. **Таблица (Table / Relation):** Базовый объект СУБД. Состоит из строк (записей) и столбцов (атрибутов). Служит для непосредственного хранения данных.
2. **Индекс (Index):** Вспомогательная структура данных (обычно сбалансированное дерево B-Tree или хэш-таблица), создаваемая для ускорения поиска строк в таблице по определенным столбцам.
   * *Плюс:* Ускорение SELECT-запросов.
   * *Минус:* Замедление операций записи (INSERT/UPDATE/DELETE), так как при каждом изменении данных нужно перестраивать индексы, а также дополнительный расход дискового пространства.
3. **Представление (View):** Виртуальная таблица, не хранящая физические данные на диске. Она представляет собой сохраненный SQL-запрос `SELECT`. Позволяет скрывать от пользователей сложные структуры таблиц и разграничивать доступ к столбцам.
4. **Хранимая процедура (Stored Procedure):** Именованный набор SQL-инструкций и управляющих конструкций (циклов, условий), компилируемый и хранящийся непосредственно в СУБД. Выполняется на стороне сервера, что снижает сетевой трафик между клиентом и БД.
5. **Триггер (Trigger):** Специальная хранимая процедура, которая автоматически запускается сервером БД при совершении определенных событий манипулирования данными (DML: `BEFORE/AFTER INSERT`, `UPDATE`, `DELETE`). Используется для обеспечения сложной бизнес-логики и аудита изменений.
6. **Ограничение (Constraint):** Правила, накладываемые на данные в столбцах для поддержания ссылочной и логической целостности:
   * `PRIMARY KEY` (Первичный ключ) — уникально идентифицирует каждую строку в таблице, не может содержать NULL.
   * `FOREIGN KEY` (Внешний ключ) — связывает таблицу с первичным ключом другой таблицы для поддержания ссылочной целостности.
   * `UNIQUE` — гарантирует уникальность значений в столбце.
   * `NOT NULL` — запрещает пустые значения.
   * `CHECK` — проверяет соответствие значения заданному логическому условию (например, `Age >= 18`).

## Вопрос 10. Основы реляционной алгебры. Синтаксические конструкции операторов структурированного языка запросов (SQL). Оператор SELECT

### 1. Основы реляционной алгебры
**Реляционная алгебра** — это теоретический процедурный язык запросов, основанный на теории множеств. Отношения (таблицы) выступают операндами, а результатом операций также являются отношения.

* **Основные операции:**
  1. **Выборка / Селекция (Selection) — обозначается $\sigma$ (сигма):** Выбирает подмножество кортежей (строк) отношения, удовлетворяющих заданному условию. Аналог в SQL: `WHERE`.
  2. **Проекция (Projection) — обозначается $\pi$ (пи):** Выбирает подмножество атрибутов (столбцов) отношения с удалением дубликатов строк. Аналог в SQL: список полей после `SELECT` (с `DISTINCT`).
  3. **Соединение (Join) — обозначается $\bowtie$:** Объединяет два отношения по общему атрибуту. Бывает естественным (по совпадающим именам полей) и тета-соединением (по условию сравнения). Аналог в SQL: `INNER JOIN`.
  4. **Объединение (Union) — обозначается $\cup$:** Объединяет все строки двух совместимых по структуре отношений. Аналог в SQL: `UNION`.
  5. **Пересечение (Intersection) — обозначается $\cap$:** Выбирает строки, присутствующие в обоих отношениях. Аналог в SQL: `INTERSECT`.
  6. **Разность множеств (Difference) — обозначается $-$:** Выбирает строки, которые есть в первом отношении, но отсутствуют во втором. Аналог в SQL: `EXCEPT` (или `MINUS`).
  7. **Декартово произведение (Cartesian Product) — обозначается $\times$:** Строит отношение, где каждая строка первого отношения скомбинирована с каждой строкой второго. Аналог в SQL: `CROSS JOIN` (или `FROM Table1, Table2`).

### 2. Синтаксические конструкции SQL
SQL (Structured Query Language) подразделяется на функциональные группы:
* **DDL (Data Definition Language):** Определение структуры данных (`CREATE`, `ALTER`, `DROP`, `TRUNCATE`).
* **DML (Data Manipulation Language):** Манипулирование данными (`INSERT`, `SELECT`, `UPDATE`, `DELETE`).
* **DCL (Data Control Language):** Управление доступом (`GRANT`, `REVOKE`).
* **TCL (Transaction Control Language):** Управление транзакциями (`COMMIT`, `ROLLBACK`, `SAVEPOINT`).

### 3. Оператор SELECT: структура и порядок выполнения
Оператор `SELECT` используется для выборки данных из таблиц.

* **Полная синтаксическая структура запроса:**
  ```sql
  SELECT [DISTINCT] список_столбцов_или_выражений
  FROM таблица_источник
    [INNER | LEFT | RIGHT | FULL OUTER] JOIN присоединяемая_таблица 
      ON условие_связи
  WHERE фильтр_для_строк
  GROUP BY столбцы_группировки
  HAVING фильтр_для_агрегированных_групп
  ORDER BY столбцы_сортировки [ASC | DESC]
  LIMIT количество_строк OFFSET смещение;
  ```

* **Логический порядок выполнения запроса сервером СУБД (Logical Query Processing Phase):**
  Интересно, что порядок написания запроса не совпадает с порядком его исполнения сервером:
  1. `FROM` — СУБД находит исходные таблицы и вычисляет декартово произведение, если таблиц несколько.
  2. `JOIN` — выполняются условия соединения `ON`, формируя единый рабочий массив данных.
  3. `WHERE` — строки исходного массива фильтруются по заданным условиям (на этом этапе еще нельзя использовать агрегатные функции, такие как `SUM`, `AVG`, `COUNT`).
  4. `GROUP BY` — отфильтрованные строки объединяются в группы по указанным столбцам.
  5. `HAVING` — сформированные группы фильтруются по условиям, которые могут содержать агрегатные функции (например, `HAVING COUNT(*) > 5`).
  6. `SELECT` — вычисляются выражения в списке выборки, удаляются дубликаты при наличии `DISTINCT`.
  7. `ORDER BY` — итоговый набор данных сортируется по указанным критериям.
  8. `LIMIT / OFFSET` — отсекаются лишние строки для пагинации.
